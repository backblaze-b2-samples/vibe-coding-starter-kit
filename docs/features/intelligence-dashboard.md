# Intelligence Dashboard

## Purpose

Surface issue intelligence from the latest (or a selected) snapshot via a Next.js frontend. The dashboard is intentionally boring: clusters with counts and labels, category bars, activity by week, and links out to GitHub for canonical text.

Design principle: someone should be able to open this Monday morning and understand the state of the backlog in under two minutes. No embedding visualizations, no fancy 3D scatter plots.

## Routes

| Route | Description |
|---|---|
| `/` (root) | Intelligence overview — latest snapshot |
| `/intelligence` | Same overview, dedicated URL |
| `/intelligence/snapshots` | List of all snapshots with status and cost |
| `/intelligence/snapshots/[id]` | Full report for a specific snapshot |
| `/intelligence/snapshots/[id]/issues` | Filterable, paginated issue list |
| `/intelligence/snapshots/[id]/clusters/[clusterId]` | Cluster drill-down with all issues |

The root `/` page is the intelligence overview (replacing the generic starter kit dashboard as specified in AGENTS.md §2).

## Components (`apps/web/src/components/intelligence/`)

| Component | Description |
|---|---|
| `ClusterCard.tsx` | Label, size badge, top 3 issue titles with GitHub links |
| `ClusterGrid.tsx` | Responsive 3-column grid of ClusterCards; unclustered rendered separately |
| `CategoryBreakdown.tsx` | Horizontal bar chart (Recharts) of issue categories |
| `ActivityTimeline.tsx` | Stacked area chart, weekly issue creation by cluster |
| `B2RoleDistribution.tsx` | Horizontal progress bars for each B2 role |
| `SpecDepthHistogram.tsx` | Bar chart of spec depth score buckets (0-2, 3-4, …) |

| `RunSnapshotButton.tsx` | Triggers POST /api/intelligence/snapshots, polls via query refetch |

## Data Layer

All fetches go through `apps/web/src/lib/queries.ts`:

| Hook | Endpoint | Notes |
|---|---|---|
| `useSnapshots()` | `GET /api/intelligence/snapshots` | Polls every 3s if any snapshot is `running` |
| `useSnapshotReport(id)` | `GET /api/intelligence/snapshots/:id` | Full dashboard payload |
| `useSnapshotIssues(id, filters)` | `GET /api/intelligence/snapshots/:id/issues` | Paginated, filterable |
| `useSnapshotClusters(id)` | `GET /api/intelligence/snapshots/:id/clusters` | Cluster list |
| `useSnapshotActivity(id)` | `GET /api/intelligence/snapshots/:id/activity` | Weekly timeline |
| `useTriggerSnapshot()` | `POST /api/intelligence/snapshots` | Mutation, invalidates `snapshots` on success |

All API functions are in `apps/web/src/lib/api-client.ts`. Query keys are in `qk.intel.*` namespace within `qk` in `queries.ts`.

## Design Rules

- Use only tokens from `apps/web/src/app/globals.css` (no new CSS variables)
- Use only primitives from `apps/web/src/components/ui/` (no new component libraries)
- Issue body is never shown in the UI — only title. Body is visible at the GitHub link.
- Issue titles are shown verbatim with a `target="_blank"` link to `html_url`
- No per-user aggregations (no "top contributor" panels)

## UX States

Each data-dependent component handles three states using the existing `ErrorState` and `EmptyState` primitives:

- **Loading**: Skeleton placeholders (same height as loaded content)
- **Error**: `ErrorState` with retry button
- **Empty**: `EmptyState` with icon and "run the pipeline" CTA

The `RunSnapshotButton` shows a spinner while the mutation is pending and toasts on success/failure via `sonner`.

## Snapshot polling

`useSnapshots()` uses TanStack Query's `refetchInterval` option to poll every 3 seconds when any snapshot has `status === "running"`. Once all snapshots are complete or failed, polling stops. The snapshot list page shows a `Loader2` icon next to running snapshots.

## Issue content privacy

The dashboard displays:
- Issue number and title (public GitHub data, shown verbatim)
- Classification metadata (category, b2_role, spec_depth_score) — generated by the pipeline
- Cluster label and summary — generated by the pipeline

It does NOT display:
- Issue body text
- Author information beyond what's needed for linking
- Any per-user aggregations

For any issue text beyond the title, users follow the GitHub link. This is explicit in the `IssueDetailPanel` as a note: "Issue body is shown on GitHub."
